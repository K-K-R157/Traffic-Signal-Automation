2. DATA ACCESS LAYER IMPLEMENTATION
====================================

2.1 DAL Architecture
--------------------

The DAL is implemented using the Repository Pattern:

Component Hierarchy:
Application Layer (API Server, Business Logic)
        |
        v
Data Access Layer (MySQLRepository, TrafficViolationLogger)
        |
        v
Database Layer (MySQL Server)

------------------

2.3 DAL Methods and Functions
------------------------------

MySQLRepository Methods:
------------------------

1. initialize()
   - Creates database tables
   - Seeds default users
   - Returns: None

2. get_user(username: str)
   - Retrieves user by username
   - Returns: dict with user data or None

3. create_user(username, password, role, display_name)
   - Inserts new user record
   - Returns: None (raises ValueError if exists)

4. list_violations()
   - Retrieves all violation records
   - Returns: list of dict

5. update_violation_action(violation_id, action_taken)
   - Updates action_taken flag
   - Returns: bool (success/failure)

6. delete_violation_if_action_taken(violation_id)
   - Deletes violation if action was taken
   - Returns: tuple (success: bool, reason: str)


TrafficViolationLogger Methods:
--------------------------------

1. _connect()
   - Establishes MySQL connection
   - Creates cursor object
   - Returns: None

2. _ensure_table()
   - Creates traffic_violations table if not exists
   - Returns: None

3. log_violation(vehicle)
   - Inserts violation record
   - Prints console message
   - Returns: None


2.4 Connection Management
--------------------------

Strategy: Connection per request (stateless)

Connection Flow:
1. Method called
2. Create new connection
3. Execute query
4. Commit transaction
5. Close connection
6. Return result

Error Handling:
- Try-except blocks for all DB operations
- Graceful degradation on failure
- Error messages logged
- No application crash


2.5 DAL Code Examples
----------------------

Example 1: Get User
-------------------
def get_user(self, username: str):
    conn = None
    cursor = None
    try:
        conn = self._connect()
        cursor = conn.cursor(dictionary=True)
        cursor.execute(
            "SELECT username, password_hash, role, display_name "
            "FROM app_users WHERE username = %s",
            (username,)
        )
        return cursor.fetchone()
    except Exception as exc:
        self._last_error = str(exc)
        return None
    finally:
        if cursor:
            cursor.close()
        if conn:
            conn.close()


Example 2: Insert Violation
----------------------------
def log_violation(self, vehicle):
    try:
        self._connect()
        self._cursor.execute(
            "INSERT INTO traffic_violations "
            "(vehicle_id, vehicle_type, side, violation_type, "
            " in_middle, out_middle) "
            "VALUES (%s, %s, %s, %s, %s, %s)",
            (
                vehicle.vehicle_id,
                vehicle.vehicle_type,
                vehicle.original_side,
                "SIDE_LANE",
                1 if vehicle.in_middle else 0,
                1 if vehicle.out_middle else 0,
            )
        )
        self._conn.commit()
    except Exception as exc:
        self._disabled = True
        print(f"[WARN] DB logging disabled: {exc}")


Example 3: Update Record
-------------------------
def update_violation_action(self, violation_id: int, action_taken: bool):
    conn = None
    cursor = None
    try:
        conn = self._connect()
        cursor = conn.cursor()
        cursor.execute(
            "UPDATE traffic_violations "
            "SET action_taken = %s WHERE id = %s",
            (1 if action_taken else 0, violation_id)
        )
        conn.commit()
        return cursor.rowcount > 0
    except Exception as exc:
        self._last_error = str(exc)
        return False
    finally:
        if cursor:
            cursor.close()
        if conn:
            conn.close()


2.6 Security Measures
----------------------

1. Parameterized Queries
   - All SQL uses placeholders (%s)
   - No string concatenation
   - Prevents SQL injection

2. Password Hashing
   - PBKDF2 algorithm
   - werkzeug.security library
   - Never store plain text

3. Input Validation
   - Username length check (min 3 chars)
   - Password strength check (min 6 chars)
   - Role whitelist enforcement

4. Error Hiding
   - Generic error messages to user
   - Detailed errors logged internally
   - No stack traces exposed


2.7 DAL Benefits
-----------------

1. Separation of Concerns
   - Business logic does not contain SQL
   - Database changes do not affect business logic
   - Easy to switch databases

2. Code Reusability
   - Single method for each operation
   - Used by multiple API endpoints
   - Consistent error handling

3. Maintainability
   - All DB code in one place
   - Easy to update queries
   - Centralized connection management

4. Testability
   - Can mock repository
   - Unit test business logic separately
   - Integration tests for DAL

3.2 Data Flow Example
----------------------

User Login Flow:
1. User submits credentials via React UI
2. POST /api/auth/login endpoint called
3. API calls MySQLRepository.get_user(username)
4. DAL executes SELECT query
5. Password hash verified
6. Session token created
7. Response sent to UI


Violation Logging Flow:
1. Vehicle crosses stop line with violation
2. Intersection calls violation_logger.log_violation(vehicle)
3. DAL executes INSERT query
4. Violation ID auto-generated
5. Timestamp recorded
6. Admin sees violation in UI

5. BEST PRACTICES IMPLEMENTED
==============================

1. Connection Pooling Alternative
   - Connection per request (stateless)
   - Suitable for low-medium traffic
   - No connection leaks

2. Transaction Management
   - Explicit commit() calls
   - Rollback on exceptions
   - ACID compliance

3. Error Logging
   - Last error stored in _last_error
   - Console warnings for DB failures
   - No silent failures

4. Resource Cleanup
   - Finally blocks for cursor/connection close
   - No dangling connections
   - Proper garbage collection

5. Code Documentation
   - Docstrings for all methods
   - Inline comments for complex logic
   - Clear naming conventions



6. FUTURE ENHANCEMENTS
======================

1. Connection Pooling
   - Use mysql.connector.pooling
   - Reduce connection overhead
   - Better for high traffic

2. ORM Integration
   - Consider SQLAlchemy
   - Object-relational mapping
   - Less SQL boilerplate

3. Caching Layer
   - Redis for frequently accessed data
   - Reduce database load
   - Faster response times

4. Read Replicas
   - Separate read/write databases
   - Load balancing
   - Better scalability

5. Migration Tools
   - Alembic for schema changes
   - Version control for database
   - Rollback capabilities
